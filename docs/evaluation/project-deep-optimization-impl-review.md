## P0-2 收紧认证边界，停止默认信任裸 `X-User-Id`

### 综合评分

综合评分：**86 / 100**
评级：**B+，放行**
一句话评价：**默认身份边界已经收紧，开发/测试 fallback 被显式配置隔离，仍需后续用真实 e2e 覆盖跨用户资源访问。**

### 评分总览

| 维度 | 分数 | 评价 |
| --- | ---: | --- |
| 计划符合度 | 90 / 100 | 覆盖 Guard、用户解析、前端默认用户、测试/灰度显式开关。 |
| 范围守纪 | 86 / 100 | 聚焦身份封口；仅额外接入 `/auth/me` 启动校验，属于计划内前端登录态收紧。 |
| 代码质量 | 84 / 100 | 复用现有 Guard、decorator、controller 分层；controller 中配置读取略重复，但比引入新抽象更克制。 |
| 测试覆盖 | 82 / 100 | 新增 Guard/util 单测，更新前端认证头测试；未本地跑依赖 DB/Redis 的 e2e。 |
| 主流对齐度 | 88 / 100 | 生产默认 token 认证、测试显式 fallback，符合常见后端身份边界做法。 |
| 风险闭环 | 86 / 100 | 灰度与测试通过环境变量显式保留旧 fixture 能力；真实跨用户资源隔离待 P1 闭环测试补强。 |

### 一致点 / 亮点

- `AuthGuard` 不再在缺少 token 时默认放行业务接口。
- `resolveUserId()` 默认不接受裸 header，消除了 controller 层误信任 header 的默认路径。
- 前端不再生成生产默认 demo 用户，未登录会进入登录页。
- 测试/灰度继续可用，但必须显式配置 `AUTH_ALLOW_HEADER_USER_ID=true`。

### 偏差 / 不足

- 计划中的“灰度和测试 fixture 改为真实 token 或显式开启测试 fallback”本步选择显式 fallback，真实 token fixture 暂不做，避免扩大改动面。
- 暂未实现“用户 A 无法通过 header 读取用户 B 资源”的真实 HTTP e2e；当前通过默认拒绝 header + service 既有隔离逻辑降低风险。
- controller 内部读取 `auth.allowHeaderUserId` 有少量重复，后续若认证策略继续扩展，可再沉淀为专门的身份解析服务。

- 无。

### 结论

- [x] 放行（综合分 ≥ 75 且无 P0）
