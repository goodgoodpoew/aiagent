## 一致点 / 亮点

- `ChatContextService` 继续负责用户消息落库、附件事实关联和缓存读取；LLM payload 选择交给 `ContextBuilderService`。
- 预算足够时保留全部候选消息，避免短会话行为退化。
- 预算不足时从最新消息往前构造连续上下文块，保持短期语境和原顺序。
- 失败消息过滤、`metadata.parts` 文本投影继续复用既有 `toLlmMessages()`。
- 日志只输出计数、估算 token 和裁剪状态，不输出用户内容。

## 偏差 / 不足

- 计划中提到 `recentMessageLimit`，实现中将其作为调试参数保留，实际裁剪主要由 token 预算决定；这是可接受偏差，因为预算足够保留全部消息更符合现有短会话体验。
- 单条最新消息超过预算时仍会超过 `maxPromptTokens`，实现选择保留一条避免空 payload；后续需要内容级截断或文件/工具结果预算拆分。
- `maxPromptTokens=6000` 仍是固定默认值，没有接入模型能力声明。
- 未做真实上游模型请求验证，当前验证停留在单元测试与 build。
