## 一致点 / 亮点

- 定位从“能请求模型”升级为“AI Agent 工作台底座”，能解释现有后端运行时、工具、文件、模型供应商和流式协议为什么值得继续投入。
- 明确了产品边界：不做普通 ChatGPT 壳，也不短期冲企业级 AI 平台，避免范围膨胀。
- 里程碑顺序合理：先产品化与可信会话，再结构化运行态和工具，然后推进文件知识、多模型 adapter 和可部署能力。
- 验收标准强调用户可观察行为，例如会话恢复、工具过程回放、文件读取事实、provider/model/usage 追溯。
- 与仓库既有规划兼容，没有推翻 `aiagent.stream.v2`、`MessageFile`/`SessionFile`、Agent Runtime 和工具网关方向。

## 偏差 / 不足

- 本次是上层路线图，不包含具体代码修改位置、DTO、组件和测试文件级别的拆分；M0/M1 开发前必须进入 `docs/plans/` 做小步计划。
- 里程碑没有绑定日期，这是刻意取舍；当前项目更需要先固定顺序和验收，再根据个人开发节奏排期。
- 身份认证方案只定义了目标，没有在本文件中选定 JWT、session cookie 或开发态 mock user 的具体方案，应在 M1 计划中决策。
- RAG、MCP、多模型 adapter 都被放到后续阶段，短期不会立刻解决“看起来更像 Agent”的展示诉求，但这能降低返工风险。
