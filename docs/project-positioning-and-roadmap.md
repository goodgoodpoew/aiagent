# 项目定位与里程碑路线图

生成日期：2026-06-06

## Summary

当前项目不应该继续定位为“能请求大模型的聊天 Demo”，而应该明确升级为：

> 面向个人与小团队知识工作的 **AI Agent 工作台底座**。

它的核心价值不是提供一个新的聊天框，而是把模型供应商、会话记忆、文件上下文、结构化流式协议、工具调用、Agent Runtime、可观测与可部署能力组合成一个可持续演进的 AI 应用平台。前端是工作台入口，后端是 Agent 运行与治理中枢，数据库与 Redis 承担状态、任务和可恢复事件。

一句话定位：

> 让用户围绕会话、文件和工具完成可追踪、可恢复、可扩展的 AI 工作流，而不是只向模型发送一次 prompt。

## 为什么当前定位尴尬

项目已经超过了简单 Demo 的技术形态，但还没有形成稳定产品定位：

- 前端仍保留 `antdXStudy` 的学习项目痕迹，示例页、真实业务页和实验能力并存，用户第一眼难以判断它到底是组件学习仓库、聊天工具，还是 Agent 平台。
- 后端已经具备会话、消息、文件、模型供应商、Redis、队列、统一响应、SSE v2、Agent Runtime、工具网关等骨架，但这些能力还没有被一个清晰产品目标统一起来。
- 当前主链路的用户价值仍表现为“把问题发给大模型并返回答案”，文件、工具、运行时、可恢复事件还没有真正形成工作流闭环。
- 身份、权限、配置、部署、可观测和测试仍处于开发态，导致项目看起来像“工程能力很多，但产品可信度还不够”。

因此，后续不宜继续堆聊天功能，也不宜突然膨胀成通用 AI 平台。更稳的路线是：以聊天为入口，以文件和工具为上下文，以 Agent Runtime 为执行核心，逐步做成一个可用、可信、可扩展的工作台底座。

## 项目应有定位

### 产品定位

本项目应定位为 **AI Agent 工作台底座**，第一批目标用户是开发者、研究者、知识工作者和小团队内部使用者。

核心场景：

- 围绕一个主题建立长期会话，而不是一次性问答。
- 上传文件、引用文件、让模型明确知道本轮实际读取了什么。
- 选择不同模型供应商和模型，在统一协议下获得一致体验。
- 使用内置工具或后续 MCP 工具完成确定性任务。
- 查看消息的生成状态、失败原因、工具过程、文件读取结果和最终产物。
- 在可部署、可测试、可观测的后端服务上持续扩展能力。

### 技术定位

前端 `antdXStudy` 不应再是组件学习项目的主叙事，而应成为 **Agent Workspace Web Client**。

后端 `ai-proxy-server` 不应只是代理服务，而应成为 **Agent Runtime Server**：

- 对外提供会话、消息、文件、模型、工具和运行流接口。
- 对内统一 provider adapter、上下文构建、工具执行、状态投影、消息持久化和失败恢复。
- 用 SSE v2 协议承载文本、思考、工具、文件读取、错误和完成事件。

### 与常见产品形态的区别

| 形态 | 本项目是否追求 | 原因 |
| --- | --- | --- |
| 普通 ChatGPT 壳 | 否 | 只做聊天框会浪费已有的文件、工具、运行时和后端工程能力。 |
| 企业级全功能 AI 平台 | 暂不追求 | 认证、审计、多租户、计费、权限矩阵会显著扩大范围。 |
| 低代码 Agent 搭建器 | 暂不追求 | 当前更需要先把运行时和工作流事实链路做稳。 |
| 个人/小团队 Agent 工作台 | 是 | 与当前架构最契合，能逐步承接文件、工具、RAG、多模型和部署能力。 |

## 主流实现对照

- 主流做法：成熟 AI 工作台通常不会让前端直接理解上游模型协议，而是通过应用层消息 parts、运行态事件、工具调用状态、文件引用事实和 provider adapter 做归一化；Agent 能力也通常先从确定性工具、可追踪运行过程和上下文治理开始，而不是一开始就开放复杂自主循环。
- 本项目现状：已经有 v2 流式协议、`MessageFile` / `SessionFile` 事实区分、模型供应商注册表、Agent Runtime、工具网关、Redis/BullMQ 和规划文档，但产品入口、身份权限、运行过程展示、RAG、可观测、部署和测试闭环仍不足。
- 本次取舍：把定位收敛为“Agent 工作台底座”，优先补产品化基础、结构化运行态、文件/工具上下文和可部署可靠性；不追求短期做成企业平台，也不重新引入旧 OpenAI-like SSE 累积格式。

## 产品北极星

项目后续的北极星指标不应是“能接入多少模型”或“页面有多少功能”，而应是：

> 用户能否在一个会话中，带着文件和工具，稳定完成一项可追踪的 AI 工作。

对应的可观察结果：

- 用户知道当前会话使用了哪个模型、读取了哪些文件、调用了哪些工具。
- 失败时能看到明确状态，并能重试、恢复或继续。
- 历史会话能还原完整上下文，不丢消息、不丢工具过程、不误导用户。
- 新增 provider、工具或文件处理能力时，不需要重写聊天主链路。

## Key Changes

后续路线应围绕六条主线推进：

1. 产品壳收敛：把前端从学习项目改造成真实工作台入口。
2. 身份与配置：从硬编码 demo 用户和 localhost 直连，升级为可部署配置与可信用户上下文。
3. 结构化运行态：把 SSE v2、消息 parts、错误状态、工具过程和持久化统一起来。
4. 文件与知识：从“附件文本拼 prompt”升级为可预算、可检索、可解释的上下文系统。
5. 工具与 Agent：从确定性内置工具开始，逐步接入 MCP 和多步运行。
6. 工程可信度：测试、可观测、灰度、部署、健康检查、权限边界。

## 重要里程碑

### M0：定位收敛与 Demo 痕迹清理

目标：让项目从命名、入口、路由、文档和本地运行体验上不再像大型 Demo。

关键工作：

- 明确主入口为 `/ai/chat`，示例页移到开发/示例分组，避免和主产品争夺叙事。
- 为前端补一层统一 API client，收口 `BASE_URL`、响应 envelope、错误处理和用户上下文。
- 清理硬编码 `localhost:3001`、固定 demo `USER_ID`、本地调试上报等开发残留。
- 更新 README 与 docs 索引，说明项目定位、启动方式、能力边界。
- 保留 Ant Design X 示例，但不让它成为项目主线。

验收标准：

- 新人打开项目文档后能在 5 分钟内说清它是 AI Agent 工作台底座。
- 主业务路径不依赖散落的硬编码后端地址和 demo 用户 ID。
- 示例页与主业务页在导航和文档中边界清楚。

### M1：产品化基础与可信会话

目标：把“能聊天”升级为“可信的多会话工作台”。

关键工作：

- 引入开发态认证与真实用户上下文，至少替换可伪造的裸 `X-User-Id` 信任模型。
- 建立用户、会话、消息、文件、模型配置的权限边界。
- 梳理会话生命周期：创建、重命名、软删除、恢复、失败消息重试、重新生成。
- 补齐流式失败、断连、刷新恢复、重复 requestId 的用户可见行为。
- 统一前端消息状态机：sending、streaming、completed、failed、cancelled。

验收标准：

- 一个用户无法读取或修改另一个用户的会话与文件。
- 刷新页面后能恢复会话、消息、附件状态和未完成/失败状态。
- 失败消息可以明确展示原因并支持重试或重新生成。

### M2：结构化消息与流式运行态完成

目标：让 v2 协议成为项目唯一主业务协议，支持后续思考、工具、文件和引用。

关键工作：

- 完成消息 `parts` 的前后端一致模型，`content` 仅作为文本投影。
- 统一 `message.part.*`、`message.completed`、`stream.completed`、`stream.failed` 的 reducer 和持久化行为。
- 支持 reasoning / thinking 摘要作为独立 part，而不是混入正文。
- 支持文件读取 part、工具调用 part、错误 part 的展示和历史回放。
- 清理旧 OpenAI-like SSE 主路径，保留必要 legacy 示例说明。

验收标准：

- 前端主聊天页不再依赖上游 `choices/delta` 结构。
- 历史消息能回放文本、文件读取、工具状态和错误状态。
- 新增一种 part 类型时，不需要重写整个聊天流解析器。

### M3：工具化与 Agent Runtime 闭环

目标：让项目真正从聊天应用进入 Agent 工作台阶段。

关键工作：

- 将文件读取等确定性工作迁入工具体系，作为内部工具执行。
- 建立工具注册、工具权限、工具执行超时、工具错误和工具结果持久化规范。
- 前端展示工具调用过程：参数、运行中、成功、失败、结果摘要。
- 接入第一批实用内置工具，例如文件读取、会话内搜索、简单代码/文本处理。
- 为 MCP 预留但不急于全面开放，先保证内置工具链路可控。

验收标准：

- 一轮会话能产生可见、可追踪、可持久化的工具执行过程。
- 工具失败不会污染最终消息状态，用户能理解失败原因。
- 新增内置工具只需实现 adapter/registry，不侵入聊天 controller。

### M4：文件知识库与上下文治理

目标：让文件能力从“读文本给模型”升级为“可管理的知识上下文”。

关键工作：

- 文件解析异步化，支持 parsing、ready、failed 等状态推送。
- 引入 token 预算器，按模型上下文窗口裁剪历史消息、附件文本和工具结果。
- 支持文件摘要、分块、引用片段和回答来源。
- 视复杂度引入向量检索或轻量全文检索，先服务会话级知识，再扩展到用户级知识库。
- 区分“文件归档在会话中”和“本轮模型实际读取内容”的 UI 展示。

验收标准：

- 大文件不会阻塞上传请求。
- 模型上下文中实际放入了哪些文件片段可解释、可回放。
- 回答能展示引用来源或至少展示本轮使用的文件片段摘要。

### M5：多模型 Provider Adapter 与模型治理

目标：从 OpenAI-compatible 代理升级为真正的多供应商模型运行层。

关键工作：

- 定义 provider adapter 接口：请求构建、流解析、非流式响应、错误归一化、usage 解析。
- 完成 OpenAI-compatible adapter 的标准化，再逐步接入 Anthropic、Gemini 或本地模型网关。
- 模型配置加入能力声明：是否支持工具、reasoning、图片、文件、最大上下文、价格、限流。
- 前端模型选择器展示能力差异，而不是只展示模型名称。
- 记录每条 assistant 消息的 provider、model、usage、成本估算和能力快照。

验收标准：

- 新接入一个 provider 不需要改前端协议和消息 reducer。
- 不支持工具或 reasoning 的模型不会在 UI 中展示不可用能力。
- 消息历史能追溯当时使用的 provider/model/usage。

### M6：可部署、可观测、可维护

目标：让项目具备长期运行和团队协作维护的底线能力。

关键工作：

- 完善配置校验、环境分层、Docker Compose、生产构建和部署文档。
- 健康检查覆盖数据库、Redis、队列、模型 provider 可用性。
- 增加 metrics、结构化日志、requestId/traceId 串联、关键失败告警。
- 建立后端 unit/integration/e2e、前端 unit/component/e2e/visual 的最小 CI 矩阵。
- 管理接口增加权限保护、审计记录和危险操作防护。

验收标准：

- 新环境能按文档稳定启动。
- CI 能覆盖核心链路：登录/身份、会话、流式聊天、文件上传、工具调用、模型配置。
- 关键故障能在日志或指标中定位，不依赖手动猜测。

## 推荐执行顺序

第一阶段先做 M0 + M1。原因是项目当前最痛的是“定位不像产品”和“可信用户上下文不足”。如果直接推进复杂 Agent，会继续叠加在 demo 壳上，后面还要返工。

第二阶段做 M2 + M3。原因是结构化消息和工具运行态必须成对出现：没有 parts，工具过程无法自然展示；没有工具，parts 的价值又只停留在协议层。

第三阶段做 M4 + M5。原因是文件知识和多模型 adapter 都依赖前面的运行态与上下文治理，过早做会让协议和存储频繁变动。

第四阶段贯穿推进 M6。工程可信度不是最后才补，但可以按风险递增逐步加强：先测试和配置，再健康检查和可观测，最后补权限审计和生产化部署。

## Interface

后续所有里程碑应围绕这几个稳定接口演进：

- 前端工作台接口：会话、消息、文件、模型、工具的统一 API client。
- 流式协议接口：`aiagent.stream.v2`，只传应用层事件，不暴露 provider 原始流。
- 消息事实接口：`Message.content` 是文本投影，`metadata.parts` 或后续 parts 表是结构化事实。
- 工具接口：工具注册、工具输入 schema、执行结果、错误、权限、审计。
- Provider adapter 接口：把不同模型供应商归一成统一运行态事件和 usage。

## Implementation Notes

- 不要把“Agent”理解为一开始就让模型自主规划一切。第一版 Agent 工作台应优先做好确定性工具、运行过程可见、失败可恢复。
- 不要为了多模型而让前端理解多套 provider 协议。provider 差异必须在后端 adapter 层消化。
- 不要把所有文件内容无脑拼进 prompt。文件上下文需要事实记录、预算、截断、摘要和引用。
- 不要用示例页数量证明项目成熟。成熟度来自主工作流是否稳定、可恢复、可测试。
- 不要提交 `.env`，不要手动修改 `antdXStudy/src/.umi/`。

## Test Plan

本文件本身是路线图，不直接触发代码测试。后续每个里程碑需要在拆分计划中定义自己的验证组合：

- 前端基础改造：`pnpm test:unit`、`pnpm test:components`、`pnpm build`。
- 后端运行态改造：`pnpm test:unit`、`pnpm build`，涉及数据库/Redis 时补 `pnpm test:integration`。
- 关键用户流：使用 Playwright 覆盖登录/会话/文件/流式聊天/工具调用。
- 协议变更：新增协议 fixture 和 reducer 单元测试，确保历史消息兼容。

## Assumptions

- 项目短期目标是个人或小团队内部可用，不直接进入企业多租户 SaaS。
- 主聊天链路只维护 v2 协议，不回退到旧 OpenAI-like SSE 累积格式。
- PostgreSQL、Redis、BullMQ、Prisma、Umi、Redux Toolkit 和 Ant Design X 继续作为当前技术底座。
- 文件存储短期保留本地实现，中长期再抽象到对象存储。
- 现有 docs 中的流式结构化、文件工具化、测试集成、灰度测试等规划继续有效，本文件作为上层路线图统筹它们。
