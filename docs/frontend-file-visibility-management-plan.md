# 前端文件可见性与临时文件管理计划

生成日期：2026-06-03

## Summary

目标是在不改变“每轮消息只引用用户明确选择的文件”规则下，解决两个问题：

- 刷新或切换会话后，当前会话上传过的文件仍可见。
- 增加一个临时统一文件管理页，让用户知道自己上传过什么、哪些文件属于哪些会话、哪些消息实际引用过文件。

## Key Changes

- 后端补齐只读查询接口：
  - `GET /api/files`：分页查询当前用户全部未删除文件，支持 `purpose/status/sessionId` 筛选。
  - `GET /api/sessions/:id/files`：查询当前会话归档文件，即 `SessionFile`。
  - 返回统一文件 VO：`id/name/type/extension/size/status/purpose/url/createdAt/updatedAt`，附带 `sessionCount/messageCount`。
- 前端新增文件领域封装：
  - 新增 `src/service/file.ts`，封装文件列表、会话文件、删除、下载入口。
  - 新增 Redux 文件状态，区分 `sessionFiles` 与 `content.attachments`。
  - `content.attachments` 只表示“当前输入框本轮待发送附件”，刷新后不自动恢复。
- 聊天页改造：
  - 上传成功后保持现有体验：文件进入输入框附件区，并调用会话关联接口归档。
  - 初始化、刷新、切换会话时加载 `GET /api/sessions/:id/files`，在聊天区展示“会话文件”列表。
  - 会话文件只做可见归档；用户点击“引用/添加”后，才进入本轮输入框附件区并随下一条消息发送。
  - 历史消息继续展示消息级附件标签，明确“这条消息实际引用了哪些文件”。
- 新增临时文件管理页：
  - 路由：`/ai/files`，菜单文案：`文件管理`。
  - 页面使用表格展示当前用户全部文件，包含文件名、状态、大小、用途、上传时间、下载、删除。
  - 支持按状态筛选，至少包含 `ready/uploaded/failed`。
  - 删除走现有软删除接口，删除后刷新列表；已在历史消息中引用过的文件不从历史消息标签中消失，只是不能再下载或再次引用。

## Test Plan

- 刷新恢复：在 `/ai/chat` 上传文件后刷新页面，输入框附件为空，但“会话文件”列表能看到该文件。
- 明确引用：从“会话文件”点击引用后，文件进入输入框附件区，发送消息时 `fileIds` 包含该文件。
- 不误引用：刷新后直接发送新问题，不携带历史会话文件。
- 管理页：`/ai/files` 能分页展示全部上传文件，筛选状态、下载、软删除后列表状态正确。
- 历史消息：发送时带附件的用户消息，刷新后仍显示该消息实际引用的文件标签。
- 构建检查：前端运行 `pnpm build`，后端运行 `pnpm build`。

## Assumptions

- 第一版采用“只显示归档，不自动待发送”的刷新策略。
- 文件管理页第一版覆盖当前用户全部文件，不做知识库、RAG、跨用户共享或复杂复用流程。
- `SessionFile` 仍只表示会话归档，`MessageFile`/消息 metadata 才表示某条消息实际引用了哪些文件。
- UI 文案、代码注释、计划文档均使用简体中文。
