# 前端界面设计规范

生成日期：2026-06-06

## 1. 适用范围

本规范适用于 `antdXStudy` 主业务前端界面，优先覆盖：

- `/ai/chat`：AI 聊天工作区、会话列表、消息流、输入区、附件与 reasoning 控制。
- `/ai/models`：模型供应商、凭据、模型配置等双栏详情与表格管理界面。
- `/ai/files`：文件列表、筛选、状态、下载和删除操作。
- `/login`：登录与注册入口页。

本规范不优先约束 `src/pages/example/` 下的 Ant Design X 组件示例页。示例页可以保留学习和演示属性，但新增真实业务页面必须遵守本文。

目标风格为**专业工作台**：克制、清晰、信息密度适中，优先服务重复操作、状态识别和长时间使用，不做营销化首页或强装饰性视觉。

## 2. 设计目标

### 2.1 统一

- 页面外壳、内容容器、表格、表单、状态标签、消息区和操作按钮必须使用同一套视觉 token。
- 禁止在不同页面各自发明颜色、圆角、阴影和间距。
- 页面标题、主操作、次操作、危险操作的位置和优先级必须稳定。

### 2.2 可适配

- 桌面端优先支持 1366px 至 1920px 宽度。
- 平板端以 768px 宽度作为关键验收点。
- 移动端以 375px 宽度作为最低验收点。
- 所有页面必须避免横向溢出；数据表格可以在表格区域内部横向滚动。

### 2.3 可维护

- 优先通过 Ant Design `ConfigProvider`、Ant Design X `XProvider`、CSS class 和 CSS 变量控制样式。
- 页面内不应散落重复的 `#f0f0f0`、`#fafafa`、`#e6f4ff` 等硬编码颜色。
- 行内样式只允许表达运行时动态布局，固定视觉样式应沉淀到 class、组件或 token。

### 2.4 适合 AI 工作台

- 聊天、文件、模型配置是一个连续工作流，而不是孤立 demo。
- AI 消息、工具过程、文件引用、错误和加载状态必须可解释。
- 界面应强调当前上下文：当前会话、当前模型、当前文件引用、当前发送状态。

## 3. 全局视觉基准

### 3.1 色彩

推荐建立以下语义 token，后续可映射到 AntD token 与 CSS 变量：

| 语义 | 建议值 | 用途 |
| --- | --- | --- |
| `colorPrimary` | `#1677ff` | 主按钮、选中态、关键聚焦 |
| `colorBgLayout` | `#f6f8fb` | 应用内容区背景 |
| `colorBgContainer` | `#ffffff` | 卡片、面板、表格容器 |
| `colorBgElevated` | `#ffffff` | Modal、Dropdown、Message |
| `colorBorder` | `#e6eaf0` | 容器边框 |
| `colorBorderSecondary` | `#f0f2f5` | 分割线、轻量边框 |
| `colorText` | `#1f2329` | 主文本 |
| `colorTextSecondary` | `#646a73` | 次文本、辅助说明 |
| `colorTextTertiary` | `#8a9099` | 弱提示、占位 |
| `colorSuccess` | `#52c41a` | 成功、就绪、启用 |
| `colorWarning` | `#faad14` | 预警、deprecated、需关注 |
| `colorError` | `#ff4d4f` | 失败、危险、删除 |
| `colorProcessing` | `#1677ff` | 上传中、解析中、流式生成中 |

使用规则：

- 选中态使用主色浅底，不使用高饱和大面积填充。
- 危险操作只在按钮、图标或确认弹窗中明确标红，不把整行大面积染红。
- 状态色必须和文字或图标同时出现，不能只依赖颜色表达状态。
- 页面背景保持浅灰，主要工作面板保持白底，避免大面积渐变或装饰色块。

### 3.2 字体与字号

- 默认字体沿用 AntD 系统字体栈。
- 正文默认 `14px`，行高不低于 `22px`。
- 页面标题使用 `20px` 或 AntD `Title level={4}`。
- 面板标题使用 `14px` 至 `16px` 加粗。
- 表格辅助信息、时间、文件大小使用 `12px`，但颜色对比度必须可读。
- 不使用随视口宽度缩放的字号。

### 3.3 间距

使用 4px 网格：

| token | 值 | 用途 |
| --- | --- | --- |
| `spaceXXS` | `4px` | 图标和短文本间距 |
| `spaceXS` | `8px` | 紧凑控件内间距 |
| `spaceSM` | `12px` | 小面板 padding、按钮组 |
| `spaceMD` | `16px` | 面板 padding、区块间距 |
| `spaceLG` | `24px` | 页面内容 padding |
| `spaceXL` | `32px` | 大区块间距 |

使用规则：

- 桌面内容区默认 padding 为 `24px`。
- 移动端内容区默认 padding 为 `12px` 至 `16px`。
- 同一页面内的面板间距保持一致，双栏布局推荐 `16px` gap。
- 表单项之间使用 AntD 默认垂直节奏，不额外叠加零散 margin。

### 3.4 圆角、边框与阴影

- 常规面板圆角使用 `8px`。
- 按钮、输入框、Tag 优先使用 AntD 默认圆角或 `6px`。
- 页面主体不使用大圆角浮层堆叠；只有独立面板、Modal、Popover 可以形成框。
- 常规容器使用 `1px solid colorBorder`。
- 默认不使用重阴影；登录面板、浮层和 Message 可使用轻阴影。

### 3.5 密度

- 管理页以扫描效率为优先，表格使用 `size="middle"` 或默认密度。
- 聊天页消息区保持阅读舒适，消息气泡最大宽度需要限制。
- 工具栏、筛选区和行操作按钮应紧凑排列，但触控目标不小于 `32px`。

## 4. 应用外壳

### 4.1 全局布局

应用外壳由侧边栏和内容区组成：

- 侧边栏桌面宽度建议 `200px` 至 `220px`。
- 内容区占满剩余宽度，最小宽度必须为 `0`，避免子元素撑破布局。
- 内容区背景使用 `colorBgLayout`。
- 页面内部面板使用 `colorBgContainer`。

桌面端推荐结构：

```text
Layout
  Sider: 产品名、主导航、用户区
  Content: 页面内容
```

移动端推荐结构：

```text
Layout
  Top navigation: 产品名、当前模块、用户操作
  Content: 纵向页面内容
```

移动端可以先采用侧边栏纵向堆叠，但后续改造应优先收敛为顶部导航或抽屉导航。

### 4.2 导航

- 一级导航表示模块，例如 `AI`。
- 二级导航表示页面，例如 `Chat 聊天`、`模型管理`、`文件管理`。
- 当前选中项使用主色浅底加主色文字。
- 图标使用 `@ant-design/icons`，同类导航图标风格保持一致。
- 导航文案必须简短，避免中英混排过长；已有约定可以保留，如 `Chat 聊天`。

### 4.3 页面头部

业务页应有稳定的页面头部：

- 左侧：页面标题和必要说明。
- 右侧：主操作和刷新、筛选等辅助操作。
- 标题区域高度不应随数据加载状态剧烈变化。
- 主操作按钮放在右上角，使用 `type="primary"`。
- 次操作使用默认按钮或图标按钮。

聊天页可以把当前会话标题放在聊天主面板头部，不强制额外页面级标题。

### 4.4 Message 与 Toast

- 全局 Message 应出现在顶部居中，但不能遮挡关键页面标题超过短时反馈范围。
- 同一失败不要连续弹出多条重复 Message。
- 复杂错误应在页面内展示 Alert 或错误态，Message 只做轻提示。
- 网络错误、加载供应商失败、SSE 失败等状态应有页面内可恢复入口。

## 5. 页面模板

### 5.1 聊天工作区

聊天页由会话列表和聊天主面板组成。

桌面端：

- 左侧会话栏宽度建议 `280px`，可在 `260px` 至 `320px` 调整。
- 右侧聊天主面板使用三段布局：头部、消息滚动区、输入区。
- 消息滚动区是唯一主要滚动容器，页面外层不应出现双重滚动。
- 宽屏下聊天内容不应无限拉宽，消息气泡应限制最大宽度。

移动端：

- 会话列表和聊天面板纵向排列。
- 会话列表不应占据过高首屏，建议后续改为可折叠区、抽屉或顶部会话切换。
- 输入区始终在聊天面板底部，不能被横向溢出的控件挤压。

会话列表：

- 单条会话包含图标、标题、更新时间和删除操作。
- 当前会话使用浅主色背景。
- 删除操作必须阻止触发会话切换，并通过 Popconfirm 二次确认。
- 标题和时间必须 ellipsis，不能撑破列表。

聊天主面板：

- 头部显示当前会话标题、刷新文件等辅助操作。
- 会话文件区使用紧凑 Tag 展示，可引用文件必须有明确按钮。
- 消息空态使用简短提示，不放大面积插画。
- 用户消息靠右，助手消息靠左。
- Markdown、表格、代码、长单词必须在气泡内部换行或横向滚动，不能撑破页面。

输入区：

- 附件列表位于发送框上方。
- reasoning 控制使用紧凑控件，移动端允许换行。
- 发送、附件等图标按钮必须有 `aria-label` 或 `title`。
- 流式生成中禁用会改变本轮上下文的控件。

### 5.2 双栏详情页

适用于模型管理等“左列表、右详情”页面。

桌面端：

- 左栏宽度建议 `300px` 至 `340px`。
- 右侧详情区使用 `minmax(0, 1fr)`。
- 左侧列表项显示名称、标识、状态和关键统计。
- 右侧详情头部显示实体名称、状态 Tag、编辑和危险操作。

移动端：

- 左右栏改为纵向排列。
- 左侧列表高度应受控，避免详情内容被推到过深位置。
- 后续可升级为“列表页进入详情页”或 Drawer 详情模式。

### 5.3 表格管理页

适用于文件管理、凭据列表、模型列表等数据管理界面。

- 页面头部包含标题、筛选和刷新。
- 表格放在白底边框容器中。
- 表格列必须设置合理宽度，长文本列使用 ellipsis。
- 操作列固定在右侧或保持稳定宽度。
- 下载、编辑、删除等行操作优先使用图标按钮；危险操作必须二次确认。
- 表格横向滚动只发生在表格容器内。
- 空列表使用 `Empty`，并提供可执行操作时给出按钮。

### 5.4 登录页

登录页可以比工作台页面更聚焦，但仍使用全局 token：

- 页面背景使用 `colorBgLayout`，可以有极轻的主色氛围，但不使用强渐变。
- 登录面板宽度建议 `360px` 至 `400px`。
- 面板内标题、Segmented、表单和主按钮垂直排列。
- 登录和注册切换时表单高度变化应稳定，不造成按钮跳动过大。
- 错误信息优先用表单校验和 Message，避免弹窗。

## 6. 组件规范

### 6.1 Button

- 主按钮：页面主创建、提交、发送等关键动作。
- 默认按钮：刷新、编辑、筛选触发、次级确认。
- 文本或链接按钮：表格行内低风险操作。
- 危险按钮：删除、禁用、撤销等不可直接恢复操作，必须配合确认。
- 纯图标按钮必须有 `aria-label` 或 `title`。
- 同一操作区最多一个主按钮。

### 6.2 Tag 与状态

状态标签使用稳定语义：

| 状态 | 颜色 | 文案示例 |
| --- | --- | --- |
| 成功 | `success` | 已就绪、启用、通过 |
| 处理中 | `processing` | 上传中、解析中、生成中 |
| 失败 | `error` | 解析失败、校验失败 |
| 警告 | `warning` | Deprecated、需配置 |
| 默认 | `default` | 未校验、禁用、未配置 |
| 强调 | `gold` 或 `blue` | 默认、预置 |

要求：

- 同一业务状态在不同页面使用同一文案。
- 文件状态、模型状态、凭据状态不能各自写一套相互冲突的标签。
- Tag 内容应短，不承载长说明；长说明放 Tooltip 或详情文本。

### 6.3 Table

- 每个表格必须有稳定 `rowKey`。
- 重要文本列使用 `ellipsis`，必要时配合 Tooltip。
- 时间列统一展示本地时间，后续可抽取格式化工具。
- 操作列宽度固定，避免按钮数量变化导致表格抖动。
- 加载时使用 Table 自带 loading。
- 空态文案要说明当前筛选条件下无数据。

### 6.4 Modal 与 Form

- Modal 标题使用动作加对象，例如 `新增供应商`、`编辑模型`。
- 表单使用 `layout="vertical"`。
- 必填项必须有明确校验文案。
- JSON 字段需要在提交前校验，错误必须定位到对应字段或给出明确 Message。
- 关闭 Modal 时应清理编辑态和表单状态。
- 宽表单 Modal 建议 `width={640}`，普通表单使用默认宽度。

### 6.5 Empty、Loading 与错误态

- Empty 文案必须说明当前状态，例如 `暂无会话`、`请选择或新增模型供应商`。
- 页面级加载使用 `Spin` 或骨架，但不能隐藏已有可读内容过久。
- 局部刷新只在对应按钮或局部容器显示 loading。
- 错误态必须提供可恢复动作，例如刷新、重试、重新选择。
- SSE 失败、上传失败、解析失败等异步错误必须保留在界面中，不能只弹 Message 后消失。

### 6.6 消息气泡

- 用户气泡和助手气泡位置固定，不能因为内容类型变化而左右跳动。
- 气泡最大宽度建议桌面端不超过消息区的 `72%`，移动端不超过 `86%`。
- Markdown 表格在气泡内部横向滚动，代码块保留等宽字体。
- 超长单词、URL、文件名必须断行或省略。
- 工具过程、reasoning 摘要、文件引用应与最终回答形成清晰层级。

### 6.7 文件与附件

- 文件名必须 ellipsis，不得撑破 Tag、表格或附件条。
- 文件大小统一格式化为 `B`、`KB`、`MB`。
- 上传中、已上传、已就绪、解析失败必须有明确图标或状态标签。
- 会话文件表示归档可见，输入区附件表示本轮待发送，两者必须视觉区分。
- 失败文件不可再次引用，禁用原因应可理解。

### 6.8 输入控件

- Select、Switch、InputNumber 等紧凑控件在工具栏中使用 `size="small"`。
- 表单中的输入控件使用默认尺寸。
- 移动端工具栏控件允许换行，但不得互相遮挡。
- Placeholder 只放示例或格式提示，不替代 label。

## 7. 响应式规则

### 7.1 断点

| 断点 | 宽度 | 规则 |
| --- | --- | --- |
| 移动端 | `< 768px` | 单列、纵向堆叠、内容 padding 收缩 |
| 平板 | `768px - 1023px` | 优先单列或紧凑双栏，关键控件换行 |
| 桌面 | `1024px - 1599px` | 标准工作台双栏或表格布局 |
| 宽屏 | `>= 1600px` | 增加内容最大宽度或优化栏宽，避免空白失衡 |

### 7.2 移动端底线

- 页面主体宽度不得超过视口宽度。
- 表格可以横向滚动，但页面不能整体横向滚动。
- 按钮文本不能溢出按钮；必要时使用图标按钮或换行布局。
- 重要操作不能因为换行掉到不可见区域。
- 输入区和发送按钮必须始终可用。

### 7.3 宽屏规则

- 管理页内容最大宽度建议 `1280px` 至 `1440px`。
- 聊天页可以占满宽度，但消息内容需要限制阅读宽度。
- 左侧列表不随宽屏无限扩大。

## 8. 工程约定

### 8.1 主题入口

后续改造建议在 `antdXStudy/src/app.ts` 中统一接入：

```tsx
<ConfigProvider theme={{ token: appThemeToken }}>
  <XProvider>
    {container}
  </XProvider>
</ConfigProvider>
```

要求：

- AntD 和 Ant Design X 必须共享同一套主题意图。
- 主题 token 文件建议放在 `src/theme/`。
- 页面组件不直接感知底层色值，只使用组件、class 或 token。

### 8.2 CSS 组织

推荐逐步沉淀这些基础 class 或组件：

- `app-shell`：应用外壳。
- `workspace-page`：页面内容容器。
- `workspace-header`：页面头部。
- `workspace-panel`：白底边框面板。
- `workspace-toolbar`：筛选和操作工具栏。
- `status-tag`：业务状态标签封装。
- `management-table`：管理表格容器。
- `chat-workspace`：聊天工作区。
- `chat-input-panel`：聊天输入区。

### 8.3 样式禁止项

- 不在页面内重复硬编码同一颜色。
- 不使用大面积渐变、装饰图形或营销式 hero 布局作为业务页主体。
- 不把多个卡片互相嵌套成复杂阴影层。
- 不让固定格式元素因为 hover、loading、长文本而改变尺寸。
- 不使用负 letter-spacing。

### 8.4 文案

- UI 文案、代码注释、提交信息使用简体中文。
- 按钮文案使用动词加对象，例如 `新增供应商`、`刷新会话`。
- 状态文案使用短语，例如 `已就绪`、`解析失败`。
- 错误文案说明用户下一步能做什么，避免只写 `Network Error`。

## 9. 后续落地改造清单

### 9.1 第一阶段：主题和外壳

- 在 `src/app.ts` 增加 `ConfigProvider`，统一 AntD token。
- 新增 `src/theme/`，沉淀颜色、圆角、间距和阴影 token。
- 改造 `src/layouts/index.tsx` 和 `src/layouts/index.css`，减少行内样式。
- 统一侧边栏、内容区背景、导航选中态和用户区。

### 9.2 第二阶段：基础页面组件

- 抽取页面头部组件，统一标题、说明、主操作和工具栏布局。
- 抽取白底边框面板组件，替代各页面重复容器样式。
- 抽取业务状态 Tag 映射，统一文件、模型、凭据状态。
- 抽取管理表格容器，统一滚动、边框、空态和操作列风格。

### 9.3 第三阶段：聊天体验

## 10. 验收清单

新增或改造页面必须逐项检查：

- 页面是否使用统一背景、面板、边框、圆角和间距。
- 页面头部、主操作、次操作、危险操作是否位置稳定。
- 所有加载、空、错、禁用、处理中状态是否可见且可恢复。
- 长标题、长文件名、长 URL、Markdown 表格和代码块是否不撑破布局。
- 移动端 375px 是否无整体横向滚动。
- 平板 768px 是否没有控件重叠。
- 宽屏 1920px 是否没有主要内容过度稀疏或阅读线过长。
- 图标按钮是否具备 `aria-label` 或 `title`。
- 状态是否不只依赖颜色表达。
- UI 文案是否为简体中文。

建议验证命令：

```bash
cd antdXStudy
pnpm test:components
pnpm test:visual
```

建议视觉验收视口：

- `375 x 812`：移动端。
- `768 x 1024`：平板。
- `1366 x 768`：常规桌面。
- `1920 x 1080`：宽屏桌面。

## 11. 当前基线问题记录

当前界面已具备基础可用性，但存在以下需要通过后续改造解决的问题：

- 全局主题入口较薄，`XProvider` 和 AntD token 尚未形成统一主题。
- 页面内行内样式较多，颜色、边框、padding 和圆角分散。
- 聊天页移动端主要是纵向堆叠，会话列表容易占据首屏。
- 宽屏下聊天区阅读线和空白控制不足。
- Message 可能在顶部遮挡当前页面信息，复杂错误缺少页面内恢复入口。
- 模型管理和文件管理的表格、面板、状态标签尚未复用统一组件。

以上问题不要求在本文档落地时立即修复；它们是后续 UI 规范实施的优先改造依据。
