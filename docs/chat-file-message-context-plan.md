# 本地文件进入大模型上下文的会话/消息关联改造计划

生成日期：2026-06-03

## 1. 目标

当前阶段不建设完整文件资产中心、知识库、RAG、跨会话文件库，只解决一个核心问题：**用户在聊天页面上传的本地文件，能够被当前轮大模型可靠读取**。

核心原则：

- 文件模块保持独立，只负责上传、存储、解析、读取和删除。
- 消息决定本轮模型实际读取哪些文件。
- 会话只做文件归档聚合，方便历史展示和后续扩展。
- 当前 v1 不自动把整个会话历史文件注入每轮请求。

一句话边界：**文件独立存储，消息决定模型读什么，会话只负责归档。**

---

## 2. 数据模型

### 2.1 UploadedFile

`UploadedFile` 仍是独立文件资产表，不增加会话业务语义。

它负责保存：

- 文件名、MIME、扩展名、大小。
- 存储 key、hash、下载 URL。
- 解析状态、解析文本、解析 metadata。
- 文件归属用户和软删除状态。

### 2.2 SessionFile

`SessionFile` 记录某个会话中出现过哪些文件。

职责：

- 支撑空状态直接上传文件时创建并选中新会话。
- 支撑未来会话文件面板、文件归档、RAG 范围选择。
- 不代表每轮请求都会自动携带这些文件。

约束：

- `sessionId + fileId` 唯一，保证幂等。
- 删除会话时级联删除关联，不删除原始文件资产。

### 2.3 MessageFile

`MessageFile` 记录某条消息实际携带了哪些文件，是本轮模型上下文的事实来源。

职责：

- 精确还原“这条消息发送时模型看了哪些文件”。
- 避免同一会话多轮上传不同文件后互相污染上下文。
- 为后续审计、引用来源、token 统计、RAG 命中结果预留结构。

约束：

- `messageId + fileId` 唯一，保证幂等。
- 删除消息时级联删除关联，不删除原始文件资产。

## 3. 后端流程

### 3.1 上传文件

上传接口只负责文件资产本身：

```text
POST /api/files/upload
  -> 校验文件类型和大小
  -> 写入本地存储
  -> 尝试解析文本内容
  -> 保存 UploadedFile
  -> 返回 fileId、状态和元数据
```

上传模块不感知会话，也不主动创建会话。

### 3.2 直接上传文件

前端上传完成后，如果当前没有会话：

```text
POST /api/sessions
body: { title, fileIds }
```

后端创建会话，并通过 `SessionFile` 记录这些文件在会话中出现过。

### 3.3 发送消息

发送消息时，前端只提交当前输入框中 `ready` 状态附件的 `fileIds`。

后端处理规则：

- 校验文件属于当前用户且未删除。
- 只读取当前请求携带的 `fileIds`。
- 可读文件写入 `MessageFile`，并注入本轮 LLM 上下文。
- 所有合法文件写入 `SessionFile`，用于会话归档。
- 不可读文件写入消息 metadata 的不可用说明，但不注入模型。

关键语义：

```text
SessionFile = 这个会话出现过哪些文件
MessageFile = 这条消息实际让模型读取了哪些文件
```

## 4. 前端流程

### 4.1 无会话直接上传文件

```text
选择文件
  -> 上传文件
  -> 如果没有当前会话，创建并选中新会话
  -> 附件继续保留在输入框
```

用户随后输入问题并发送时，文件才作为本轮消息附件进入模型上下文。

### 4.2 有会话上传文件

```text
选择文件
  -> 上传文件
  -> 追加到当前输入框附件区
  -> 调用会话文件关联接口做归档
```

不立即生成消息，不触发模型请求。

### 4.3 发送消息

```text
点击发送
  -> 只提交 ready 附件 fileIds
  -> 后端保存 MessageFile / SessionFile
  -> 请求完成后清空附件
```

下一轮如果用户没有重新选择文件，则不会自动复用上一轮文件。

---

## 5. 上下文规则

v1 明确规则：

- 当前消息有附件：只携带当前消息附件内容。
- 当前消息无附件：不自动携带历史文件。
- 会话历史文件只作为归档，不参与当前 v1 的自动上下文注入。
- 上传成功但解析不可用的文件，不进入模型上下文，并需要在 UI 或消息 metadata 中可解释。

推荐注入格式：

```text
用户随消息附带了以下文件内容，请只在相关时引用：

<file id="..." name="需求说明.md" type="text/markdown">
...解析后的文本内容...
</file>
```

---

## 6. 重复会话修复

前端允许使用 `draft-*` 会话做乐观展示，但必须在收到后端真实 `sessionId` 后合并：

```text
draft-* 会话
  -> 收到 X-Session-Id
  -> 删除 draft 实体
  -> 插入真实 session
  -> 迁移本地消息的 sessionId
```

禁止简单 `upsert` 真实会话，否则等待接口返回时会短暂出现两个同名会话。

---

## 7. 暂不做事项

v1 明确不做：

- 跨会话文件库。
- 独立文件管理页面。
- 知识库、RAG、向量检索。
- 文件版本管理。
- 自动引用整个会话历史文件。
- 复杂空会话生命周期状态。

允许暂时存在“有文件但没有消息”的空会话，后续可通过 `draft/active` 状态或清理任务优化。
