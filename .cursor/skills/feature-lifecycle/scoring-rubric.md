# 锐评打分量表

两道闸门共用的评分规范。每个维度打 0–100 分，给出综合分（默认按权重加权，可简单平均）、评级（A/B/C/D）和一句话评价。格式对齐 `docs/evaluation/first-milestone-project-evaluation.md`。

**放行线（共用）**：综合分 ≥ 75 且无 P0 致命项即放行。牢记第一原则——达标即放行，不为提分而过度设计；功能符合最初预期且思路贴近主流即可。

评级参考：A ≥ 90 / B+ 80–89 / B 75–79 / C 60–74（需修订）/ D < 60（重做）。

---

## 闸门A · 计划锐评维度

目的：检验计划的**可执行程度**与**距离主流实现的差距**。

| 维度 | 权重 | 看什么 |
| --- | ---: | --- |
| 可执行性 | 25% | 步骤是否清晰、可落地、边界明确；能否照着直接开发。 |
| 范围控制 | 15% | 是否最小可用、第一版范围克制；有无过度设计或镀金。 |
| 主流对齐度 | 20% | 核心思路（数据流/分层/协议/错误处理）是否贴近市面主流与框架最佳实践。 |
| 项目契合度 | 15% | 是否复用既有架构与约定（v2 协议、应用服务层、统一响应、模型供应商注册表等），不另起炉灶。 |
| 风险识别 | 10% | 是否识别关键风险、依赖、回滚/兜底。 |
| 验收明确性 | 15% | 验收标准是否可验证、可观察、可测试。 |

**P0 致命项（命中则不放行）**：
- 计划与项目核心约定冲突（如重新引入旧 OpenAI-like SSE 累积格式、私改 `src/.umi`、计划提交 `.env`）。
- 关键步骤无法落地或缺少验收标准。
- 范围严重失控，无法在可控上下文内推进且未拆分。

## 闸门B · 实现锐评维度

目的：审查当前实现相对**计划阶段**的偏差与不足。

| 维度 | 权重 | 看什么 |
| --- | ---: | --- |
| 计划符合度 | 30% | 实现与计划是否一致；偏差是否被记录与解释。 |
| 范围守纪 | 15% | 有无超范围实现或漏实现；是否守住"本步只做一件事"。 |
| 代码质量 | 20% | 符合项目约定与分层、可读、命名清晰、无仅复述代码的注释。 |
| 测试覆盖 | 15% | 关键链路是否有单元/集成测试；lint/build 是否通过。 |
| 主流对齐度 | 10% | 实现思路是否仍贴近主流，未在落地时跑偏。 |
| 风险闭环 | 10% | 计划中识别的风险是否处理或显式标注遗留。 |

**P0 致命项（命中则不放行）**：
- 破坏既有能力（如影响旧端点/旧链路、回归既有测试）。
- 实质性偏离计划且无任何说明。
- 引入未修复的 lint/build 失败，或提交了不应提交的产物/密钥。
