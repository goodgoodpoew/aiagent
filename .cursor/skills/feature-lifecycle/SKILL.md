# 功能全流程开发（feature-lifecycle）

本 skill 规定在 `aiagent` 仓库内，把一个功能从想法推进到落地的标准流程。目标是让每个功能：**设计有依据、计划可落地、实现不跑偏、交付可验证**。

## 第一原则（必须时刻记住）

> 水满则溢，月盈则亏，世界上没有真正完美的系统架构，更没有完美的技术实现方案。我要求每一个功能的实现只要符合最初的预期，并且跟市面上主流实现的思想差不多即可。不必追求完美。

落地含义：
- 评审打分的目标是**发现致命偏差与硬伤**，不是凑满分。达标即放行，禁止为了刷分过度设计。
- 「与主流思想差不多」指核心思路一致（如数据流、分层、协议、错误处理范式），不要求与某个大厂实现逐行对齐。
- 当继续打磨的边际收益很低时，明确写下"已达预期、暂不深化"并进入下一阶段。

## 全流程概览

复制此清单跟踪进度（每个功能一份）：

```
功能全流程进度：
- [ ] 阶段1 功能设计：明确预期、边界、非目标
- [ ] 阶段2 对照主流实现：调研市面主流方案与本项目框架最佳实践
- [ ] 阶段3 制定计划：写入项目 docs/（必须落地为文件）
- [ ] 闸门A 计划锐评打分：多维度评分，达标才放行（硬性要求1）
- [ ] 阶段4 拆分计划：若计划过大，拆为独立可执行小步（写入 docs/plans/<feature>/）
- [ ] 阶段5 开发实现：按计划逐步实现
- [ ] 闸门B 实现锐评打分：审查与计划的偏差（硬性要求2）
- [ ] 阶段6 测试集成 + 验证：测试通过、构建通过、行为符合预期
```

两道闸门（闸门A、闸门B）是**硬性要求**，不可跳过。

---

## 阶段1 功能设计

把"想法"收敛成"可判定的预期"。产出至少回答：
- **要解决什么问题**：一句话描述用户/系统价值。
- **预期行为**：成功长什么样（可观察、可验证）。
- **边界与非目标**：本次不做什么，避免范围蔓延。
- **影响面**：涉及前端 `antdXStudy` / 后端 `ai-proxy-server` / 数据库 / 协议 的哪些部分。

不确定需求时，用 `AskQuestion` 让用户在离散选项中选择，不要自行猜测大方向。

## 阶段2 对照主流实现

在动手前，对照"市面主流怎么做"和"本项目框架最佳实践怎么做"。
- 本项目栈：前端 Umi Max 4 + Ant Design X 2.x + Redux Toolkit；后端 NestJS + Prisma + PostgreSQL + Redis；聊天链路只走 `aiagent.stream.v2` 协议。优先复用既有分层（`ConversationApplicationService`、会话生命周期、消息持久化、统一响应信封、模型供应商注册表、流式 v2 编排）。
- 需要时用 `WebSearch` 查证主流方案（注意用当前年份检索）。
- 输出"主流做法 vs 本项目现状 vs 本次取舍"的简短对照，作为计划的设计依据。
- 切忌脱离项目现有约定另起炉灶（参考 `AGENTS.md` / `CLAUDE.md` 的开发约定）。

## 阶段3 制定计划（必须落地为项目文件）

计划必须写成项目内的文件，不能只停留在对话里。遵循现有约定：

- **总计划**：`docs/<feature>-plan.md`。
- 结构参考现有计划（如 `docs/token-usage-estimation-plan.md`）：`Summary / Key Changes / Interface / Implementation Notes / Test Plan / Assumptions`，可按需增减。
- 计划要明确：复用了哪些现有模块、不改什么（如不改 Prisma schema、不引第三方依赖）、第一版覆盖范围。
- 所有文档、注释、提交信息用简体中文（项目硬性约定）。

写完总计划后，**立即进入闸门A**。

---

## 闸门A 计划锐评打分（硬性要求1）

每次计划制定完成后，必须触发锐评打分机制，从多维度检验**可执行程度**与**距离主流实现的差距**。

1. 按 [scoring-rubric.md](scoring-rubric.md) 的「计划锐评」维度逐项打分（0–100）。
2. 把锐评写成项目文件：`docs/evaluation/<feature>-plan-review.md`（参考 `docs/evaluation/first-milestone-project-evaluation.md` 的评分总览 + 综合评分 + 评级 + 一句话评价格式）。
3. 判定：
   - **综合分 ≥ 75 且无 P0 致命项** → 放行，进入阶段4/5。
   - **综合分 < 75 或存在 P0 致命项** → 修订计划后重新打分，直到达标。
4. 牢记第一原则：达标即放行，不要为提分而过度设计。

## 阶段4 拆分计划（计划过大时必须拆分）

当计划过大、上下文难以精准聚焦时，必须拆分。

**拆分原则**：每一步小实现必须**独立**，即使有互相引用也要限定在一个**可控范围**内（一步只动一个清晰的关注点，依赖前序步骤的产物而非与之交织）。

**落地方式**（遵循 `docs/plans/stream/` 现有范式）：
- 目录：`docs/plans/<feature>/`。
- 索引：`docs/plans/<feature>/README.md`，包含建议执行顺序与依赖关系图。
- 每步一个文件：`NN-<step-name>.md`（如 `01-...`、`02-...`）。

**每一步小计划必须完整包括以下字段**（缺一不可）：

```
修改位置、目的、动机、修改原因、实施方案、产出、验收、风险与注意事项
```

使用 [plan-template.md](plan-template.md) 作为每步小计划的模板。每个字段的实施方案/产出/验收建议用 `- [ ]` 勾选项，便于执行时标注进度（参考 `docs/plans/stream/02-backend-text-stream-v2.md`）。

> 拆分后的每一步同样适用闸门B：每完成一步实现，就对该步做一次实现锐评打分。

## 阶段5 开发实现

- 按计划/拆分步骤逐步实现，一次聚焦一步，避免跨步混改。
- 严格遵守项目约定：`@/` 指向 `src/`、聊天链路只维护 v2 协议、不手改 `src/.umi/`、不提交 `.env`、注释用中文、不写仅复述代码的注释。
- 实现过程中同步把对应小计划里的 `- [ ]` 勾为 `- [x]`，并在文件顶部加执行标注（日期 + 实际落地情况），与现有计划文档保持一致。
- 编辑后用 `ReadLints` 检查改动文件的 lint，引入的问题要修复。

每完成一个计划单元（或一步拆分），**立即进入闸门B**。

---

## 闸门B 实现锐评打分（硬性要求2）

每次开发实现完成后，必须触发锐评打分机制，审查**当前实现相对计划阶段的偏差与不足**。

1. 按 [scoring-rubric.md](scoring-rubric.md) 的「实现锐评」维度逐项打分（0–100）。
2. 把锐评写成项目文件：`docs/evaluation/<feature>-impl-review.md`（可在同一文件按步骤追加小节）。锐评要明确列出：
   - 与计划的**一致点**与**偏差点**（实际改了什么、和计划差在哪、为什么）。
   - **不足与遗漏**（漏实现、超范围、未覆盖的风险）。
   - 是否需要回到计划修订，或开启补救小步。
3. 判定：
   - **综合分 ≥ 75 且偏差可接受** → 进入测试集成与验证。
   - **存在重大偏差或 P0 项** → 修复或更新计划后重新打分。
4. 牢记第一原则：实现符合最初预期且思路贴近主流即可通过。

## 阶段6 测试集成与验证

- **后端**（`ai-proxy-server`）：`pnpm lint`、`pnpm build`，相关 `pnpm test:unit` / `test:integration`（集成测试需 `pnpm test:env:up`）。
- **前端**（`antdXStudy`）：`pnpm test:unit` / `test:components`，必要时 `pnpm build`。
- **验证**：对照阶段1 的"预期行为"逐条确认；无法在本地真实联网验证的（如真实模型请求）要在计划/锐评中明确标注"未实地验证"，参考现有计划文档的诚实标注方式。
- 验证通过后，在总计划/锐评文档回填最终状态，关闭进度清单。
