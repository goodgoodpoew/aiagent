# 竞业达 AI 辅助开发规范（强制执行）

> **文档性质**：团队强制执行制度，所有研发人员必须遵守
> **适用对象**：产品、开发、测试全员
> **生效日期**：2026 年 3 月
> **版本**：V1.0
> **关联培训**：《AI 赋能产品全流程 — 从调研到交付的效率革命》（2026-02-26）

---

## 第一章 总则

### 1.1 目的

规范团队使用 AI 工具（Claude Code、Gemini、Codex 等）进行软件开发的全流程，确保：

1. AI 生成的代码质量可控、风格统一
2. 开发过程可追溯、可回滚
3. 敏感数据不泄露
4. 团队协作有序、责任清晰

### 1.2 适用范围

本规范适用于所有使用 AI 工具参与的研发活动，包括但不限于：

- **产品侧**：需求调研、竞品分析、PRD 撰写、用户故事生成、原型辅助设计
- **开发侧**：界面设计与前端生成、后端业务逻辑开发、Bug 排查与修复
- **测试侧**：测试用例生成、接口自动化脚本编写、Bug 复现分析、测试报告撰写
- **交付侧**：部署文档生成、客户问题排查辅助、培训材料制作、实施方案编写

### 1.3 基本原则

| 原则 | 说明 |
|------|------|
| **AI 是工具，人是负责人** | AI 生成的所有产出（代码、文档、方案），由使用者本人负责质量 |
| **先约束，再生成** | 必须通过项目约束文件（如 CLAUDE.md）或 Prompt 模板约束 AI 行为，禁止无约束裸跑 |
| **小步快跑，及时提交** | 每完成一个功能点立即 Git Commit，禁止攒一堆再提交 |
| **交叉审查，不自审** | AI 写的代码不能只让同一个 AI 审查，必须人工 + 跨模型审查 |
| **安全红线，不可触碰** | 敏感数据禁止发送给任何 AI 服务 |

### 1.4 关键环节 AI 工具强制使用清单

以下工作环节**必须**借助 AI 工具完成，但**不限定具体工具**，团队成员可从已批准的工具中自行选择：

| 环节 | 强制要求 | 可选工具 |
|------|---------|---------|
| **代码安全审查** | 提交前必须使用 AI 工具进行安全扫描 | Claude Code / Cursor / GitHub Copilot / Codex |
| **Code Review 辅助** | PR 合并前必须使用 AI 工具辅助审查 | Claude Code / Cursor / Codex / GitHub Copilot |
| **单元测试生成** | 核心业务逻辑必须有 AI 辅助生成的测试覆盖 | Claude Code / Cursor / GitHub Copilot |
| **接口文档生成** | 后端接口必须使用 AI 工具生成文档初稿 | Claude Code / Cursor / Gemini |
| **Bug 排查** | 复杂 Bug（涉及多模块）必须使用 AI 工具辅助定位 | Claude Code / Codex / Cursor |
| **需求文档结构化** | 原始需求必须使用 AI 工具整理为结构化文档 | Claude Code / Gemini / 通义灵码 |

> **说明**：本清单强制的是\"必须使用 AI 工具\"，而非\"必须使用某个特定工具\"。团队推荐 Claude Code，但不做强制绑定。

## 第二章 安全红线（不可违反）

### 2.1 禁止发送给 AI 的内容

以下内容**严禁**以任何形式（对话、文件上传、代码粘贴）发送给外部 AI 服务：

| 类别 | 具体内容 | 违规后果 |
|------|---------|---------|
| **客户真实数据** | 学生姓名、身份证号、成绩、考试数据 | 按数据泄露事故处理 |
| **认证凭证** | API Key、数据库密码、Token、SSL 证书 | 立即吊销凭证并通报 |
| **商业机密** | 合同金额、投标价格、客户名单、未公开的商务协议 | 按保密协议追责 |
| **服务器信息** | 生产环境 IP 地址、内网拓扑、运维账号 | 按安全事故处理 |

### 2.2 正确做法

- **数据脱敏**：用 Mock 数据替代真实数据再提交给 AI
- **凭证隔离**：所有密钥放在 `.env` 文件中，`.env` 加入 `.gitignore` 且不粘贴给 AI
- **本地优先**：敏感业务逻辑优先在本地环境调试，AI 仅辅助非敏感部分

### 2.3 代码提交安全检查

每次 Git Commit 前，开发者必须确认：

- [ ] 代码中没有硬编码的密码、Key、Token
- [ ] 没有引入包含真实用户数据的测试文件
- [ ] `.env` / `credentials` 类文件未被 `git add`
- [ ] AI 生成的代码中没有暴露内部服务地址

## 第三章 环境配置规范

### 3.1 通用要求

无论使用哪种 AI 工具，开发者必须确保：

1. **AI 工具版本统一**：同一项目组的成员使用相同版本的 AI 工具，避免产出差异
2. **全局约束配置**：在 AI 工具中配置全局开发规范，至少包含以下约束：
   - 使用中文注释和 commit message
   - 代码风格遵循项目 ESLint / Prettier / Checkstyle 配置
   - 生成代码时包含完整的类型定义
   - 不要生成无用的 console.log / System.out.println
   - 不要引入项目未使用的第三方依赖
   - 修改代码时只改需求涉及的文件，不要擅自重构其他代码
3. **工具安全设置**：确认 AI 工具的数据传输方式（API 调用 vs 网页上传），涉及代码的场景必须使用 API 模式

### 3.2 项目级约束文件（项目负责人必须执行）

每个项目的根目录必须包含一份 **AI 项目约束文件**，且**提交到 Git 仓库**，确保所有 AI 工具都能读取项目上下文。

> 使用 Claude Code 的项目，此文件为 `CLAUDE.md`；使用 Cursor 的项目为 `.cursorrules`；使用 Codex 的项目为 `AGENTS.md`。如果团队使用多种工具，可以同时维护多份或使用一份通用文件。

项目约束文件必须包含以下章节：

| 章节 | 必填内容 |
|------|---------|
| 项目概述 | 一句话描述项目定位 |
| 技术栈 | 框架、UI 库、构建工具、主要依赖的版本号 |
| 项目结构 | 目录树及各目录职责说明 |
| 命名规范 | 文件命名、变量命名、路由路径、CSS 类名等约定 |
| 常用命令 | dev / build / test / lint 等命令 |
| 业务约定 | 角色类型定义、权限模型、数据流向等业务规则 |

### 3.3 Claude Code 配置示例（推荐）

如果团队选择使用 Claude Code，以下为推荐配置：

**① 全局 `settings.json`（`~/.claude/settings.json`）**

```json
{
  "permissions": {
    "allow": ["Bash", "Edit", "Write", "Read", "WebFetch", "NotebookEdit"],
    "deny": [],
    "ask": []
  },
  "model": "opus"
}
```

**② 全局 `CLAUDE.md`（`~/.claude/CLAUDE.md`）**

按 3.1 中的全局约束内容配置即可，模板见附录 C。

**③ MCP 工具配置**

项目中使用的 MCP 工具在项目根目录的 `.mcp.json` 中统一配置，提交到 Git 仓库，确保团队成员环境一致。

### 3.4 其他 AI 工具配置说明

> **能力分层说明**：目前市场上 AI 编程工具能力差异显著。团队经过评估，将工具分为以下层级：
>
> | 层级 | 定义 | 工具 |
> |------|------|------|
> | **🔴 生产级** | 具备深度推理、跨文件理解、自主 Agent 能力，可独立完成复杂业务开发 | **Claude Code (Opus)**（团队主推）、**Codex 5.3**（最佳备选）、Kiro (Opus) |
> | **🟡 工程级** | 具备良好的代码生成和补全能力，适合常规开发任务 | Cursor、GitHub Copilot |
> | **🟢 辅助级** | 以代码补全和简单生成为主，适合轻量任务 | 通义灵码、Trae、文心快码 Comate、豆包 MarsCode |
>
> *注：工具的能力层级取决于其底层模型。Kiro 接入 Opus 模型后具备生产级能力；Cursor 若配置 Opus 模型也可用于 A 级任务。未配置高级模型的工具按其默认模型定级。*
>
> A 级任务（见第十二章）**必须**使用生产级工具完成。

| 工具 | 项目约束文件 | 配置方式 |
|------|------------|---------|
| **Cursor** | `.cursorrules` | 放在项目根目录，格式与 CLAUDE.md 类似 |
| **Kiro (AWS)** | `steering rules` | Kiro IDE 内置配置，支持 MCP |
| **Codex (OpenAI)** | `AGENTS.md` | 放在项目根目录，描述项目规范和约束 |
| **GitHub Copilot** | `.github/copilot-instructions.md` | 放在 `.github` 目录下 |
| **Gemini** | 无标准约束文件 | 通过 Prompt 模板（见第十章）约束 |
| **通义灵码**（阿里云） | 无标准约束文件 | 通过工具内置设置 + Prompt 模板约束 |
| **Trae**（字节跳动） | `.traeRules` | 放在项目根目录，免费 AI IDE |
| **文心快码 Comate**（百度） | 无标准约束文件 | 通过工具内置设置约束 |
| **豆包 MarsCode**（字节跳动） | 无标准约束文件 | 通过工具内置设置约束，免费 |

#### 4.2.3 产品文档审核要点

AI 生成的产品文档**尤其容易出现以下问题**，产品经理必须重点审核：

- [ ] **编造功能**：AI 可能会"合理推测"产品应该有某个功能，但实际上并不在规划中
- [ ] **竞品信息过时**：AI 训练数据有截止日期，竞品的最新功能可能不准确
- [ ] **数据引用失实**：AI 引用的市场数据、政策文件可能是编造的，**必须逐条验证来源**
- [ ] **需求镀金**：AI 倾向于把需求写得很"完美"，加入不必要的功能点，导致范围蔓延
- [ ] **忽略业务约束**：AI 不了解公司资源、客户预算、交付周期等现实约束

#### 4.2.4 禁止行为

- 禁止将客户的真实需求文档（含客户名称、项目金额等）直接发送给 AI
- 禁止将 AI 生成的 PRD 不经审核直接发给客户或开发团队
- 禁止用 AI 生成的竞品数据作为投标文件依据（必须人工验证）

### 4.3 测试工程师 AI 使用指引

#### 4.3.1 适用场景

| 场景 | AI 职责 | 人工职责 |
|------|--------|---------|
| **测试用例生成** | 根据需求文档或接口文档生成测试用例（正向、逆向、边界） | 审核用例覆盖度，补充业务特殊场景 |
| **接口自动化脚本** | 根据接口文档生成自动化测试脚本 | 调试运行、调整断言值、确认 Mock 数据合理性 |
| **Bug 分析辅助** | 分析错误日志、堆栈信息，定位可能的原因 | 确认根因，验证修复方案 |
| **测试报告生成** | 根据测试结果数据生成结构化测试报告 | 审核数据准确性，补充风险评估和建议 |
| **回归测试规划** | 根据代码变更范围，建议需要回归的测试范围 | 确认回归范围，补充经验判断 |

#### 4.3.2 推荐工具

| 任务 | 主推 | 其他可选 |
|------|------|---------|
| 测试用例生成 | **Claude Code** / Cursor | GitHub Copilot（模式化任务，中级模型即可） |
| 接口自动化脚本 | **Claude Code** / Cursor | —（中级及以上模型） |
| Bug 日志分析 | **Claude Code** / **Codex** | —（复杂 Bug 建议使用高级模型） |
| 前端 UI 测试辅助 | Chrome DevTools MCP | 手动测试 |
| 测试报告 | Claude / Gemini | 通义灵码 |

#### 4.3.3 测试用例审核要点

AI 生成的测试用例**常见以下问题**，测试工程师必须重点审核：

- [ ] **覆盖面看似全面实则遗漏关键场景**：AI 擅长生成"标准"用例，但容易遗漏业务特有的边界条件
- [ ] **断言值不准确**：AI 不了解真实业务数据，生成的预期结果可能有误
- [ ] **Mock 数据不合理**：AI 生成的测试数据可能不符合实际业务规则（如身份证校验位、学号格式）
- [ ] **忽略前置条件和数据依赖**：AI 可能假设测试环境有某些数据，实际并不存在
- [ ] **重复覆盖、遗漏覆盖**：AI 可能生成大量相似用例却遗漏某个关键分支

#### 4.3.4 测试数据安全

- 测试用例中**严禁使用真实学生数据**（姓名、身份证号、成绩），必须用脱敏数据或 Mock 数据
- AI 生成的自动化脚本中不得包含真实环境的 URL、账号密码
- 测试报告中的截图、日志需检查是否包含敏感信息后再提交

### 4.4 交付/实施工程师 AI 使用指引

#### 4.4.1 适用场景

| 场景 | AI 职责 | 人工职责 |
|------|--------|---------|
| **部署文档生成** | 根据项目技术栈生成部署手册初稿（环境要求、安装步骤、配置说明） | 在实际环境验证步骤，补充客户特定配置 |
| **实施方案编写** | 根据项目需求生成实施方案框架（里程碑、资源需求、风险点） | 确认时间节点合理性，补充现场实际情况 |
| **客户问题排查** | 分析客户反馈的错误日志，提供可能的解决方案 | 验证方案可行性，在客户环境实际操作 |
| **培训材料制作** | 根据产品功能生成用户操作手册、培训 PPT 大纲 | 审核操作步骤准确性，补充截图和演示 |
| **FAQ 整理** | 根据历史工单整理常见问题和解决方案 | 确认解决方案有效性，更新过时内容 |

#### 4.4.2 推荐工具

> 工具能力分层详见 [第三章 3.4 节](#34-工具约束文件配置)。**核心任务优先使用生产级工具**。

| 任务 | 主推 | 其他可选 |
|------|------|---------|
| 部署文档 / 实施方案 | **Claude Code** / Cursor | Gemini（中级模型即可） |
| 客户问题排查 | **Claude Code** / **Codex** | —（需要深度分析，建议 Opus） |
| 培训材料 | Claude / Gemini | 通义灵码 |
| PPT 制作辅助 | Gemini Canvas | — |

#### 4.4.3 交付文档审核要点

AI 生成的交付文档**常见以下问题**，交付工程师必须重点审核：

- [ ] **环境信息编造**：AI 可能编造服务器配置要求、端口号、依赖版本，必须与实际项目核实
- [ ] **步骤遗漏或顺序错误**：AI 生成的安装部署步骤可能缺少关键前置步骤
- [ ] **客户环境差异**：AI 基于通用场景生成的文档，可能不适用于客户的特定网络环境、操作系统版本
- [ ] **权限和安全配置遗漏**：AI 可能忽略防火墙规则、SSL 证书配置等安全相关步骤
- [ ] **过度理想化**：AI 倾向于给出"一切顺利"的步骤，缺少异常处理和回退方案

#### 4.4.4 交付安全红线

- **严禁**将客户现场的服务器 IP、数据库连接信息、VPN 账号等发送给 AI
- **严禁**将客户的真实业务数据（考试数据、学生信息）用于 AI 分析
- AI 生成的部署文档中如果包含示例 IP 或域名，必须替换为客户实际配置后才能使用
- 客户问题排查时，发送给 AI 的日志必须先脱敏（去除 IP、用户名、文件路径等）

## 第五章 AI 辅助开发流程规范

### 5.1 开发流程总览

```
需求文档 → 项目约束文件配置 → 方案确认（Plan Mode 等）→ 编码（小步快跑）
    → Git Commit（每步提交）→ 人工验证 → Code Review → 合并
```

### 5.2 需求输入规范

**禁止**直接口头描述需求让 AI 编码。必须提供结构化输入：

**最低要求**：Prompt 中包含以下要素

| 要素 | 说明 | 示例 |
|------|------|------|
| 功能描述 | 做什么 | "为用户管理页面添加批量导入功能" |
| 技术约束 | 用什么技术 | "使用 Element Plus Upload 组件，Excel 解析用 xlsx 库" |
| 输入输出 | 数据格式 | "上传 .xlsx 文件，解析后展示预览表格，确认后调用批量创建接口" |
| 验收标准 | 怎样算完成 | "支持 1000 行以内的 Excel，重复数据高亮提示，导入结果有成功/失败统计" |

**推荐做法**：在项目 `docs/requirements/` 目录下维护需求文档，让 AI 读文档再编码。

### 5.3 方案先行（复杂功能必须执行）

以下场景**必须**先让 AI 输出方案再编码（Claude Code 的 Plan Mode、Cursor 的 Composer 规划模式，或手动让 AI 先出方案均可）：

- 涉及 3 个以上文件的修改
- 新增功能模块
- 重构已有代码
- 修改公共组件或共享逻辑

**流程**：

1. 让 AI 输出方案（涉及哪些文件、修改思路、潜在风险）
2. 开发者审核方案，确认后再执行
3. 禁止 AI 在方案阶段直接修改代码

### 5.4 编码过程管控

#### 5.4.1 小步快跑原则

- **一个功能点 = 一次对话 = 一次 Git Commit**
- 禁止在一次对话中塞入多个不相关的需求
- 单次 AI 修改的文件数不应超过 10 个（超过时应拆分任务）

#### 5.4.2 范围控制

每次给 AI 的指令中必须明确限定修改范围：

```
✅ 正确："只修改 src/modules/school/exam/ 目录下的文件，不要动其他模块"
❌ 错误："帮我优化一下项目代码"
```

#### 5.4.3 禁止行为

| 禁止行为 | 原因 |
|---------|------|
| 让 AI 无约束地"优化"或"重构"整个项目 | 可能破坏已有功能，改动不可控 |
| 让 AI 自行决定引入新的第三方依赖 | 可能引入安全风险或许可证冲突 |
| 让 AI 修改 CI/CD 配置、部署脚本 | 影响全团队，必须人工操作 |
| 让 AI 直接操作生产环境 | 严禁，无例外 |
| 将 AI 生成的代码不经审查直接上线 | 必须 Code Review 后才能合并 |

## 第六章 Git 提交规范（铁律）

### 6.1 核心规则：AI 改完代码，立刻 Commit

```
AI 完成一个功能点 → git diff 检查改动 → git add → git commit → 人工验证
                                                                   ├── 通过 → 继续下一个
                                                                   └── 不通过 → git revert 回退
```

### 6.2 Commit Message 规范

AI 辅助开发的 Commit 必须标注 AI 工具信息：

```
<类型>: <简要描述>

<详细说明（可选）>

AI-Assisted-By: <工具名称>
Co-Authored-By: <AI 模型信息>
```

**类型标识**：

| 类型 | 说明 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `refactor` | 重构（不改变功能） |
| `test` | 测试代码 |
| `docs` | 文档 |
| `style` | 代码格式（不影响逻辑） |
| `chore` | 构建/工具链/依赖更新 |

**示例**：

```
feat: 用户管理页面添加批量导入功能

支持 xlsx 文件上传和预览，重复数据高亮提示，
导入结果展示成功/失败统计。

AI-Assisted-By: Claude Code (Opus 4.6)
```

```
fix: 修复考试列表分页查询越界问题

AI-Assisted-By: Cursor (Claude Sonnet 4)
```

```
feat: 添加成绩导出功能

AI-Assisted-By: GitHub Copilot
```

### 6.3 Commit 粒度要求

| 粒度 | 说明 | 示例 |
|------|------|------|
| **正确** | 一个功能点一个 Commit | "feat: 添加考试列表搜索功能" |
| **错误** | 多个功能混在一个 Commit | "feat: 添加搜索、分页、导出、打印功能" |
| **错误** | 整个模块一个 Commit | "feat: 完成考试管理模块全部功能" |

### 6.4 Commit 前必须检查

- [ ] `git diff` 查看所有改动，确认没有意外修改
- [ ] AI 没有擅自修改不相关的文件
- [ ] 没有遗留调试代码（console.log、debugger、TODO）
- [ ] 没有硬编码的测试数据（mock 用户名、测试密码）
- [ ] 代码能正常编译 / 构建通过

## 第七章 Code Review 规范

### 7.1 AI 代码必须 Review

所有 AI 辅助生成的代码**必须**经过 Code Review 才能合并到主分支。

**Review 分为两层**：

| 层级 | 执行者 | 关注点 |
|------|--------|--------|
| **第一层：自审** | 开发者本人 | 功能完整性、编译通过、基本逻辑正确 |
| **第二层：他审** | 团队其他成员或技术负责人 | 代码规范、安全漏洞、架构合理性、业务正确性 |

### 7.2 跨模型交叉审查（推荐）

对于核心业务逻辑，推荐使用不同 AI 模型交叉审查：

```
工具 A 编写代码 → 人工测试发现问题 → 工具 B 定位修复 → 回到工具 A 验证
```

> 示例：Claude Code 编写 → Codex 审查；Cursor 编写 → Claude Code 审查

**原因**：同一 AI 模型存在"自我确认偏差"，倾向于认为自己生成的逻辑是正确的。不同模型的"盲区"不同，交叉审查能发现更多问题。

### 7.3 Review Checklist

Code Review 时必须检查以下项目：

**功能性**：
- [ ] 功能是否符合需求描述
- [ ] 边界条件是否处理（空值、空列表、超长输入）
- [ ] 错误场景是否处理（网络异常、接口失败、数据异常）

**安全性**：
- [ ] 是否存在 SQL 注入风险
- [ ] 是否存在 XSS 风险
- [ ] 接口是否有权限校验
- [ ] 是否有硬编码的凭证或内部地址

**AI 特有问题**：
- [ ] AI 生成的注释是否准确（AI 有时会写错误的注释）
- [ ] 是否引入了不需要的依赖
- [ ] 是否重复实现了已有的公共组件或工具方法
- [ ] 硬编码的测试数据是否已清理
- [ ] AI 是否擅自修改了不相关的文件

**代码质量**：
- [ ] 命名是否符合项目规范
- [ ] 是否存在冗余代码
- [ ] 是否有未使用的变量、导入、函数

## 第八章 测试规范

### 8.1 AI 生成测试的流程

```
让 AI 读懂业务代码 → 告知测试框架和项目约定 → AI 生成测试代码
    → 开发者运行测试 → 调整 Mock 数据和断言 → 测试通过后提交
```

### 8.2 后端接口测试（必须执行）

每个后端接口**必须**有对应的测试脚本，覆盖以下场景：

| 场景 | 说明 |
|------|------|
| 正常流程 | CRUD 完整链路 |
| 参数校验 | 必填字段缺失、类型错误、超长输入 |
| 权限校验 | 未登录、越权操作 |
| 边界条件 | 空列表、分页边界、并发冲突 |
| 异常场景 | 不存在的 ID、重复创建 |

### 8.3 后端单元测试（推荐执行）

Service 层核心业务逻辑推荐编写单元测试：

- 让 AI 读取 Service 类代码后生成测试
- Mock 掉外部依赖（数据库、远程调用）
- 覆盖 if-else 分支、异常路径
- **注意**：AI 生成的测试断言值需人工确认业务正确性

### 8.4 前端测试（当前阶段）

- 核心流程由人工手动验证
- 让 AI 根据页面代码生成测试 Checklist
- 让 AI 分析页面代码中可能的交互问题（表单校验遗漏、列表为空展示等）

### 8.5 测试代码管理

- 测试代码与业务代码同步提交到 Git
- 测试用例的 Mock 数据必须使用脱敏数据
- AI 生成的测试必须先本地运行通过再提交

## 第九章 文档规范

### 9.1 AI 辅助生成文档的场景

| 文档类型 | AI 职责 | 人工职责 |
|---------|---------|---------|
| 模块技术文档 | 生成初稿（组件结构、数据流、接口说明） | 审核准确性、补充业务上下文 |
| 接口文档 | 根据代码生成接口签名和参数说明 | 确认业务含义、补充示例数据 |
| 变更日志 | 根据 Git Commit 汇总变更 | 审核分类和描述准确性 |
| 需求文档 | 将原始需求（Excel / 会议纪要）整理为结构化文档 | 确认业务正确性 |

### 9.2 文档目录约定

```
docs/
├── requirements/       # 需求文档（功能描述、验收标准）
├── architecture/       # 架构文档（AI 生成 + 人工审核）
├── api/                # 接口文档
└── changelog/          # 变更日志
```

### 9.3 文档审核要求

AI 生成的文档**必须**人工审核后才能作为正式文档使用。重点审核：

- 业务描述是否准确（AI 可能会"看起来正确但实际有误"）
- 技术细节是否与实际代码一致
- 是否有 AI 编造的内容（幻觉）

## 第十章 Prompt 管理规范

### 10.1 建立团队 Prompt 模板库

团队必须在共享仓库中维护 Prompt 模板，避免每人各自摸索：

```
prompts/
├── research/           # 调研类 Prompt 模板
│   ├── 竞品分析.md
│   └── 技术选型.md
├── development/        # 开发类 Prompt 模板
│   ├── 新增功能模块.md
│   ├── Bug修复.md
│   └── 代码重构.md
├── testing/            # 测试类 Prompt 模板
│   ├── 接口测试生成.md
│   └── 单元测试生成.md
└── review/             # 审查类 Prompt 模板
    ├── 安全审查.md
    └── 代码规范审查.md
```

### 10.2 Prompt 编写标准

所有 Prompt 模板必须包含以下结构：

```markdown
## 场景描述
（什么时候用这个模板）

## Prompt 模板
（可直接复制使用，留好占位符）

## 使用示例
（一个完整的填充示例）

## 注意事项
（使用时的注意点）
```

### 10.3 禁止的 Prompt 写法

| 禁止写法 | 原因 | 正确写法 |
|---------|------|---------|
| "帮我优化一下代码" | 范围不明确，AI 可能乱改 | "优化 UserService.java 的 queryUsers 方法，减少 N+1 查询" |
| "写一个页面" | 缺少技术约束和业务要求 | 结构化描述（见第五章 5.2） |
| "代码有 Bug，帮我看看" | 缺少上下文 | 提供报错信息 + 相关代码 + 操作步骤 |
| "帮我重构整个项目" | 范围过大，风险不可控 | 限定到具体模块或具体问题 |

## 第十一章 并行开发规范

### 11.1 并行 Agent 使用场景

部分 AI 工具（如 Claude Code、Codex）支持启动多个 Agent 并行编写代码。以下场景适合使用并行 Agent：

- 多个相互独立的功能模块同时开发
- 前端多个页面同时生成（前提是互不依赖）
- 批量生成测试代码

### 11.2 并行开发约束

| 规则 | 说明 |
|------|------|
| 互不依赖 | 并行 Agent 处理的模块之间不能有代码依赖 |
| 各自提交 | 每个 Agent 完成后独立 Commit，不混在一起 |
| 逐个验证 | 并行完成后，逐个模块验证功能正确性 |
| 冲突处理 | 如果出现文件冲突，由开发者人工解决，不让 AI 自动合并 |

### 11.3 禁止的并行场景

- 多个 Agent 同时修改同一个文件
- 多个 Agent 同时修改公共组件或共享模块
- 在没有项目约束文件的项目中使用并行 Agent

## 第十二章 AI 工具使用优先级

### 12.1 工具选择指引

> **团队主推工具**：Claude Code (Opus) — 深度推理、跨文件理解、自主 Agent 能力经团队验证，是当前唯一达到生产级的主力工具。
> **最佳备选工具**：Codex 5.3 — 具备强大的代码生成和自主执行能力，适合作为交叉审查和复杂任务的备选。
> 其他工具可用于辅助性任务，但核心业务开发**强烈建议**使用主推或备选工具。

| 场景 | 主推 | 备选 | 其他可选 |
|------|------|------|---------|
| 日常编码开发 | **Claude Code (Opus)** | Codex / Kiro (Opus) | Cursor / GitHub Copilot / Trae |
| 需求调研 / 竞品分析 | **Claude Code + Tavily MCP** | Gemini Deep Research | 通义灵码 |
| 界面设计与原型 | Gemini Canvas | Claude Code / Cursor | Trae |
| Bug 排查（复杂） | **Claude Code (Opus)** | Codex | Cursor / Kiro |
| Code Review | **Claude Code + 人工审查** | Codex 交叉审查 | 其他 AI 工具辅助 |
| 文档撰写 | Claude Code / Gemini | 通义灵码 | 文心快码 |
| 前端调试与测试 | Chrome DevTools MCP | 手动测试 | — |

### 12.2 模型分级使用制度

模型选择不做统一强制，但**按任务风险等级设定最低模型要求**。使用低于最低要求的模型完成的代码，Code Review 不予通过。

#### 🔴 A 级任务（最低要求：Opus 4.6+）

以下任务**必须**使用 Opus 模型，使用 Sonnet / Haiku 完成的产出不予合并：

| 任务类型 | 原因 |
|---------|------|
| 核心业务逻辑开发 | 涉及数据正确性、资金、权限等关键逻辑，低级模型容易出逻辑漏洞 |
| 架构设计与技术方案 | 需要深度推理和全局理解，便宜模型的方案质量明显不足 |
| 数据库设计与 SQL 编写 | 低级模型容易写出性能差或有注入风险的 SQL |
| 安全相关代码（认证、加密、权限） | 安全代码不允许"差不多就行"，必须用最强模型 |
| 复杂 Bug 排查（涉及多模块联动） | 需要长上下文理解和跨文件追踪能力 |
| 生产环境问题分析 | 给出错误判断的后果严重 |

#### 🟡 B 级任务（最低要求：Sonnet或者同级别模型）

以下任务至少使用 Sonnet，允许使用 Opus：

| 任务类型 | 原因 |
|---------|------|
| 常规功能页面开发 | 逻辑复杂度中等，Sonnet 能保证基本质量 |
| 接口联调与数据对接 | 需要理解前后端数据格式映射 |
| 单元测试 / 接口测试生成 | 测试代码需要准确的 Mock 和断言 |
| Code Review 辅助 | 需要足够的理解力发现问题 |
| 需求文档整理 | 需要准确理解业务含义 |

#### 🟢 C 级任务（允许使用 Haiku或者同级模型）

以下任务允许使用任何模型：

| 任务类型 | 说明 |
|---------|------|
| 简单的 CRUD 页面生成 | 模式固定，低级模型也能胜任 |
| 代码格式化、重命名、简单重构 | 机械性操作 |
| 文本翻译、格式转换 | 不涉及业务逻辑 |
| Commit Message 生成 | 简单摘要 |
| 注释补充 | 描述性文本 |

#### 执行机制

**不是靠禁止，而是靠结果倒逼**：

1. **Commit Message 必须标注工具和模型**：`AI-Assisted-By: Claude Code (Opus 4.6)` 或 `Cursor (Claude Sonnet 4)` 或 `GitHub Copilot`
2. **Code Review 时检查模型等级**：A 级任务用了 Haiku/Sonnet → Review 直接打回，要求用 Opus 重新生成
3. **质量问题追溯**：如果线上出现 Bug，追溯到该代码是用低级模型生成且未经充分 Review → 开发者承担责任
4. **成本与质量的平衡**：省模型费用导致的返工成本远高于模型差价，团队需建立这个共识

> **给管理者的建议**：与其争论"该用什么模型"，不如把精力放在 Code Review 关口。用便宜模型写的代码如果 Review 能过，说明任务本身就是 C 级；如果 Review 过不了反复返工，开发者自然会意识到在关键任务上省模型费用是得不偿失的。

## 第十三章 AI 工具准入管理

### 13.1 工具分级

不是所有 AI 工具都可以随意使用。按数据安全等级分为三类：

| 级别 | 定义 | 示例 | 使用要求 |
|------|------|------|---------|
| **企业级（已采购）** | 公司统一采购、已通过安全评估的工具 | Claude Code（API 付费）、Codex、Kiro、Cursor Pro、GitHub Copilot Business、公司内部部署的模型 | 可处理内部代码和非敏感业务数据 |
| **公开级（免费/个人）** | 未经公司采购的公开 AI 服务 | 免费版 ChatGPT、免费版 Gemini、通义灵码免费版、Trae、豆包 MarsCode、文心快码免费版 | **禁止**输入公司代码和业务数据，仅允许用于通用技术问答 |
| **禁止级** | 来源不明或安全性存疑的 AI 工具 | 破解版 AI 工具、非官方 API 中转服务、未知来源的本地模型 | **严禁使用** |

### 13.2 Shadow AI 治理

**Shadow AI**：员工未经批准私自使用 AI 工具处理公司数据的行为。

这是目前企业面临的最大 AI 安全隐患之一（德勤 2026 报告指出，企业发送给 GenAI 应用的敏感数据量一年内增长了 30 倍）。

**治理措施**：

1. **堵不如疏**：为团队提供好用的企业级 AI 工具（如 Claude Code、Cursor 等），减少使用野生工具的动机
2. **明确边界**：本规范 2.1 节已列出禁止发送给 AI 的内容，所有工具无一例外
3. **定期宣导**：每季度进行一次 AI 安全意识提醒
4. **发现即纠正**：发现使用未授权工具处理公司数据的，按第十四章违规处理

### 13.3 新工具引入流程

团队成员如需引入新的 AI 工具：

1. 向技术负责人提出申请，说明工具名称、用途、数据处理方式
2. 技术负责人评估数据安全风险（数据是否上传到外部服务器、服务商数据使用政策等）
3. 审批通过后添加到团队"已批准 AI 工具清单"
4. 未经审批的工具不得用于处理公司代码和业务数据

## 第十四章 违规处理

### 14.1 违规分级

| 等级 | 行为 | 处理方式 |
|------|------|---------|
| **严重** | 将客户真实数据 / 凭证发送给 AI | 通报批评 + 安全复盘 + 凭证吊销轮换 |
| **严重** | AI 代码未经 Review 直接部署到生产环境 | 通报批评 + 回滚代码 + 全面检查 |
| **一般** | 未配置项目约束文件直接使用 AI 开发 | 口头警告 + 限期整改 |
| **一般** | AI 改动后未及时 Git Commit | 口头警告 |
| **一般** | Commit Message 未标注 AI 辅助信息 | 补充标注 |
| **轻微** | 未使用 Prompt 模板，效果不佳导致返工 | 团队内部分享教训 |

### 14.2 持续改进

- 每月收集 AI 使用过程中的典型问题，更新本规范
- 每季度评估团队 AI 工具使用效率，优化流程
- 鼓励团队成员分享高质量 Prompt 和使用技巧

## 第十五章 AI 输出审核与质量兜底

### 15.1 核心原则：你提交的代码，你负责

AI 生成的所有产出（代码、文档、方案、测试），一旦开发者采纳并提交，即视为该开发者的工作成果，质量责任由开发者本人承担。"AI 写的"不是免责理由。

### 15.2 AI 输出的常见质量问题

基于行业数据（Veracode 2025 报告：约 45% 的 AI 生成代码包含安全漏洞），团队需重点关注：

| 问题类型 | 表现 | 检查方法 |
|---------|------|---------|
| **幻觉代码** | 调用不存在的 API 或方法 | 编译 / 构建验证 |
| **过时语法** | 使用已废弃的 API（AI 训练数据有截止日期） | 开启项目 lint 规则 + 查阅官方最新文档 |
| **安全漏洞** | SQL 注入、XSS、硬编码凭证 | Code Review Checklist（见第七章） |
| **许可证污染** | 生成的代码片段与 GPL 代码高度相似 | 关键模块人工审查 |
| **逻辑看似正确实则有误** | 边界条件处理不当、业务规则理解偏差 | 测试覆盖 + 人工验证核心逻辑 |
| **过度工程** | 引入不必要的抽象、设计模式、第三方库 | Review 时检查"这个复杂度是否必要" |

### 15.3 分级审核机制

| 变更风险等级 | 审核要求 |
|------------|---------|
| **高风险**（支付、权限、数据迁移、生产配置） | 必须两人 Review + 测试通过 + 技术负责人签字 |
| **中风险**（业务逻辑、接口开发、公共组件） | 至少一人 Review + 测试通过 |
| **低风险**（页面样式、文案修改、注释补充） | 自审即可，但仍需 Commit |

## 第十六章 培训与能力建设

### 16.1 AI 工具使用能力要求

所有研发人员需具备以下基础能力：

| 能力项 | 达标标准 |
|--------|---------|
| 环境配置 | 能独立完成团队指定 AI 工具的安装和配置 |
| Prompt 编写 | 能使用结构化 Prompt 获得可用的代码产出（非模糊提问） |
| 质量判断 | 能识别 AI 生成代码中的常见问题（幻觉、过时 API、安全漏洞） |
| Git 协作 | 能按规范提交 AI 辅助代码（Commit Message 标注、粒度控制） |
| 安全意识 | 清楚知道哪些内容不能发送给 AI |

### 16.2 培训计划

| 时间节点 | 培训内容 | 形式 |
|---------|---------|------|
| 新人入职首周 | 本规范宣贯 + 环境配置实操 | 1 对 1 带教 |
| 每月 | Prompt 技巧分享会 / AI 踩坑复盘会 | 团队内部分享 |
| 每季度 | AI 安全意识提醒 + 新工具/新功能培训 | 集中培训 |
| 按需 | 专项培训（AI 辅助测试、AI Debug 实战等） | 专题 Workshop |

- 高质量 Prompt 模板统一归档到团队 Prompt 库（见第十章）

- 每个项目的 AI 约束文件（CLAUDE.md / .cursorrules 等）作为团队知识资产管理
